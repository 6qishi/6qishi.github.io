为什么数据库的表名会是单数？

6qishi2025-09-182025-09-18

前几天在写代码的时候，脑子里突然冒出一个问题，一个可能很多人从未在意过的问题：数据库表名，为什么习惯用单数（比如 user），而不是复数（users）呢？
这也许是一个很多开发者凭直觉或习惯在做，却很少停下来细想的选择。这个偶然的想法让我觉得很有趣，于是我花时间深入思考了一下，并把这些想法整理成了这篇文章。

争论的起点：表的本质是什么？

我们先来思考一个哲学问题：一张数据库表，它到底是个啥？
复数派的观点很简单直接：“一张表里有很多行数据，比如用户表里有很多用户，当然应该叫 users 啦！” 听起来似乎很有道理，对吧？
但等一下，让我们用 Java 的思维方式来重新审视一下。在 Java 中，我们如何表示一群用户？通常是这样的：

看，你的集合变量名叫 userList，但集合里存放的对象的类型是 User，而不是 Users。你定义实体类的时候，也是这么写的：

没有人会把类名定义成 Users 吧？
现在，把这个思想套用到数据库上：

  • 一张数据库表，就像是 Java 里的一个 List 或 Set 集合。
  • 表名，应该描述的是这个集合中，每一行数据（每一个实体）的类型。

所以，一个存放 User 类型数据的表，理所当然应该叫做 user。它是一个 user 的集合，而不是一个叫 users 的东西的集合。

从代码的可读性出发：当 SQL 遇上 Java 思想

作为开发者，我们每天都在和代码打交道，可读性至关重要。我们来对比一下两种命名风格下的 SQL 查询：
单数风格:

读起来是不是特别顺畅？就像在 Java 代码里访问一个对象的属性：user.getName()。你的大脑几乎不需要转换，因为它完全符合我们面向对象的直觉。

复数风格:

读起来就有点别扭了。“从‘用户们’这张表里，找到‘用户们’的 ID 是 123 的那条记录，然后取出‘用户们’的名字”。感觉像是在说绕口令。

单数命名法让 SQL 更贴近自然语言和我们的编程思维。

外键命名：复数派的噩梦

如果前面的你觉得还只是“风格问题”，那外键命名这块会直接杀死比赛。这是支持单数命名法最硬核的理由。
我们还是用一个简单的场景：一个订单（Order）属于一个用户（User）。

如果用单数命名：

  • 表是 user 和 order。
  • 在 order 表里，我想加个外键指向用户，那这个字段叫什么？user_id，简直是天作之合。
  • order.user_id -> user.id
  • 清晰、直观，没有任何歧义。user_id 这个名字本身就在告诉你：“我是 user 表的 ID”。

但如果用复数命名呢？

  • 表是 users 和 orders。
  • 现在，我要在 orders 表里加一个外键… 这下有点迷惑了，该叫什么？
    • 叫 user_id？ 那字段名是单数，它引用的表 users 却是复数，逼死强迫症。
    • 叫 users_id？ 这个词读起来就很怪。“用户们的 ID”？这到底是一个用户的 ID，还是一群用户的 ID？产生了不必要的歧义。

在这个问题上，复数命名法似乎给自己制造了一个无法完美解决的小麻烦，而单数命名法则非常丝滑。

结论：选择单数，拥抱清晰

聊到这里，必须承认：表名用单数还是复数，这并不是一个能决定项目成败的重大技术问题。说实话，即使一个项目里单复数混用，只要能正常工作，大多数时候也无人在意。
但如果你能读到这里，说明你是一个愿意在细节上多做思考的开发者，这本身就是一种非常宝贵的品质。
这或许不是什么惊天动地的大问题，甚至在读这篇文章前你都未曾留意过它。但希望这篇小短文，没有浪费你的时间，并能给你带来一点点关于规范与优雅的启发。
